🎬 Persistent Terminal Sessions

PR #8: Never lose your terminal state again

The Problem

When creating demos that switch between terminal and browser, you'd lose all terminal state—working directory, environment variables, running processes. Multi-terminal workflows (server + client) were impossible.

The Solution

Persistent Sessions: Terminal state now persists across mode switches. Switch to browser and back—your terminal is exactly where you left it.

Named Sessions: Run multiple independent terminals simultaneously with @mode terminal:server and @mode terminal:client.

What This Demonstrates

  • ✓ Persistent Sessions (Issue #4): Set environment variables, switch to browser, return to terminal—state preserved
  • ✓ Named Sessions (Issue #5): Run HTTP server in terminal:server, make requests from terminal:client
  • ✓ Session Independence: Default terminal, server, and client all maintain separate state
  • ✓ Mode Switching: 7 seamless segment transitions (terminal → browser → terminal → server → client → server → terminal)

New Syntax

SyntaxDescription
@mode terminalDefault terminal session (state persists across mode switches)
@mode terminal:serverNamed session "server" (independent from other sessions)
@mode terminal:clientNamed session "client" (independent from other sessions)

Use Cases

  • ✓ Server/Client Demos: Start server, make requests from client, show both logs
  • ✓ Full-Stack Tutorials: Terminal → Browser → Terminal without losing context
  • ✓ Multi-Service Workflows: Run database, backend, and frontend in separate sessions
  • ✓ Long-Running Processes: Keep processes running while recording other parts of the demo
